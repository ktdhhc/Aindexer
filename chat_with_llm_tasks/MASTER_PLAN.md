# Chat With LLM Master Plan

## 目标与范围

- 目标：新增“基于 LLM 的文献问答/检索”能力，支持会话管理、流式回复、中断、分阶段进度反馈。
- 检索策略：先快检总索引，再按用户确认进入深度检索，必要时进入全量检索。
- 输出要求：学术场景下高结构化、一致性强、可追溯证据链（文献编号 + 原文片段/页码）。

## 总体架构

- 前端：新增 Chat 面板（会话列表 + 对话区 + 阶段进度 + 操作按钮）。
- 后端：新增 `chat` 路由与服务层工作流编排（planner/retriever/answerer）。
- 数据：新增会话与消息表，记录运行状态、阶段、取消标记、引用证据。
- Prompt：集中放入 `backend/prompts/chat/`，按阶段拆分。

## 隔离开发原则（不影响现有核心功能）

- 路由隔离：Chat 功能仅新增 `/api/chat/*`，不修改现有 `/api/index/*`、`/api/files/*`、`/api/search/*` 的对外行为。
- 数据隔离：新增 chat 专属表（会话/消息/run），不改 `documents`、`index_records`、`claims` 结构与语义。
- 运行时隔离：Chat 使用独立任务管理与取消状态，不复用索引任务 FUTURES，避免互相中断或状态污染。
- 前端隔离：Chat 区域使用独立状态对象（如 `chatState`），不复用上传/搜索已有全局状态。
- 读写边界：V1~V3 仅“读取”现有索引/文献用于检索，不改索引生成链路；如需写入仅写 chat 表。
- 发布隔离：增加 `CHAT_ENABLED` 开关，支持灰度开启，异常时可快速关闭 chat 模块且不影响主流程。

## 版本路线图

### V1 - 最小可用版（先打通闭环）

#### 目标
- 跑通从提问到流式回答的完整链路，支持会话 CRUD、流式、中断、阶段进度显示。
- 仅基于“已生成索引”做检索，不读取原文全文。

#### 功能范围
- 会话：新建、切换、删除。
- 对话：发送用户问题、流式返回回答、显示引用的 doc_id 列表。
- 中断：用户可随时取消当前生成。
- 进度：显示阶段（检索中 / 构建上下文 / 生成回答）。

#### 技术策略
- 检索：优先用现有搜索与 FTS 结果，召回 top-k 条索引上下文。
- 生成：LLM 在召回内容上回答，并给结构化输出（结论 + 证据 + 置信度）。
- 失败降级：无命中时返回“未命中 + 建议深度检索”的结构化回复（不自动深搜）。

#### 交付物
- 新增后端 chat 路由与最小数据模型。
- 前端 chat 基础 UI 与流式显示。
- Prompt 初版与输出 schema。

### V2 - 深度检索申请（用户确认后再扩大范围）

#### 目标
- 当 V1 结果“不确定/未命中”时，进入“深度检索申请”流程，由用户决定是否继续。

#### 功能范围
- 在答案中返回决策类型：`direct_answer` 或 `request_deep_search`。
- 用户确认后：对 top N 候选文献读取原文、分块分析并二次回答。
- 返回更严格证据：原文片段、页码、文献编号。

#### 技术策略
- 工作流拆两阶段：`planner`（是否需要深搜）+ `deep_reader`（分块核查）。
- 深搜任务可中断；进度显示到批次与文献粒度。
- 限额控制：单次深搜文献数、分块数、总 token 上限。

#### 交付物
- 深搜确认交互（前端弹窗/确认条）。
- 深搜执行器（后端任务状态机）。
- 深搜 prompt 与引用模板。

### V3 - 全量检索申请（分批扩展）

#### 目标
- 深搜后仍不确定时，支持“全量检索申请”，由用户选择是否继续扩展搜索。

#### 功能范围
- 深搜失败后返回 `request_full_search`。
- 用户确认后：对剩余候选文献按批次（每批 3 篇）进行检索。
- 支持早停（找到高置信命中即结束）与完整结果汇总。

#### 技术策略
- 批处理执行器：按批队列推进，记录每批结果。
- 进度细化：批次编号、已处理篇数、当前命中状态。
- 成本防护：最大批次数、最大时长、最大 token。

#### 交付物
- 全量检索确认交互与中断控制。
- 批处理后端流水线与结果聚合器。
- 失败/取消后可恢复执行（可选）。

### V4 - 稳定性与一致性增强（质量工程）

#### 目标
- 提升学术检索的稳定性、可复现性和可审计性，降低“同问不同答”的波动。

#### 功能范围
- 统一回答结构模板（结论/证据/反证/置信度/覆盖范围）。
- 引入结果缓存（相同问题 + 相同语料范围直接复用）。
- 加入回归评测集（固定问题集评估一致性与命中率）。

#### 技术策略
- Prompt 与 schema 双约束；低一致性时自动二次判定。
- 引用检查器：没有证据的结论标记为低置信或拒答。
- 日志可观测：记录每一步检索证据链与模型判定原因。

#### 交付物
- 评测脚本与基线数据。
- 一致性指标看板（命中率、引用覆盖率、重复问答一致性）。

## 关键参数建议（可在 V1 末确定）

- `N`（深度检索篇数）默认建议：`5`。
- `M`（全量补检最大篇数）默认建议：`20`。
- 每批文献数建议：`3`（与原需求一致）。
- 回答强约束建议：每条核心结论至少 1 条证据引用。

## 里程碑与验收建议

- M1（V1 完成）：可用性验收（会话、流式、中断、基础检索回答）。
- M2（V2 完成）：深搜流程验收（确认交互 + 原文分块 + 证据引用）。
- M3（V3 完成）：全量分批验收（申请、批次进度、早停策略）。
- M4（V4 完成）：质量验收（一致性、可追溯、回归测试通过）。
