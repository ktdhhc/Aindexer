# Chat With LLM V0 实施方案

## 目标

- 先打通一条最小闭环：`前端提问 -> 后端读取测试索引 -> 调用 LLM -> 返回回答 -> 前端展示`。
- V0 不追求完整检索能力，只验证链路可用性与后续可扩展性。
- 当前回答先仅基于 **一条测试索引**，跑通后再切到“全索引”或“多索引聚合”。

## 页面位置

- 新增一个 `Chat with LLM (V0)` 卡片。
- 放置位置：**搜索区和预览区下方**。
- 页面结构调整为：
  1. 上传区
  2. 搜索区 + 预览区
  3. Chat V0 区

这样用户能自然理解：
- 先上传/生成索引
- 再搜索/预览
- 最后基于索引提问

## V0 功能边界

### 包含

- 一个最小 Chat 卡片
- 一个问题输入框
- 一个发送按钮
- 一个回答展示区域
- 一个状态提示区域
- 一个后端最小问答接口
- 后端读取一条测试索引作为上下文
- 调用当前已配置的 LLM provider 回答

### 不包含

- 会话管理
- 聊天历史持久化
- 中断/取消
- SSE 流式返回（后续再加）
- 多轮对话上下文
- 全文深度检索
- 全量索引检索
- 证据链结构化引用

## 技术策略

### 1. 后端读取哪份上下文

V0 直接读取 **一条测试索引的 Markdown 内容**。

优先方案：
- 从已索引文献中取第一条（或固定一条 `doc_id`）
- 读取 `data/indexes/{doc_id}.md`
- 把这份 Markdown 全文作为上下文发给 LLM

选择 Markdown 的原因：
- 它已经是最终用户可见索引内容
- 结构清晰，最接近真实使用场景
- 不需要重新拼接字段

### 2. Prompt 策略

新增两个 Prompt 文件：

- `backend/prompts/chat_v0_system_prompt.txt`
- `backend/prompts/chat_v0_user_prompt_template.txt`

#### system prompt 职责

- 明确告诉模型：只能依据给定索引内容回答
- 不允许编造不存在的信息
- 如果索引中没有足够信息，要明确说“不确定”或“索引未提及”
- 输出中文、简洁、清楚

#### user prompt 模板结构

建议内容：

```txt
任务：根据给定的文献索引内容回答用户问题。

要求：
1. 只依据索引内容回答
2. 若索引中未提供足够信息，请明确说明“不确定”或“索引中未提及”
3. 不要编造文献中不存在的信息
4. 使用中文回答，优先给出简洁结论，再补充必要说明

用户问题：
{question}

索引内容：
{context}
```

## 后端设计

### 路由

新增：

- `backend/app/routers/chat.py`

首版接口：

- `POST /api/chat/ask_v0`

请求体：

```json
{
  "question": "这篇文献的核心结论是什么？"
}
```

返回体（V0-1 非流式）：

```json
{
  "doc_id": "doc_xxx",
  "display_name": "某篇测试文献",
  "answer": "..."
}
```

### 服务层

新增：

- `backend/app/services/chat_v0.py`

建议包含函数：

- `pick_test_chat_document() -> dict | None`
- `load_test_chat_context(doc_id: str) -> str`
- `build_chat_v0_prompt(question: str, context: str) -> tuple[str, str]`
- `run_chat_v0(question: str, provider: str, model: str | None = None) -> dict`

### Provider 调用

现有 `ProviderClient.generate_json()` 不适合 V0 问答。

建议新增文本接口：

- `ProviderClient.generate_text(...) -> str`

能力要求：
- 用普通 chat completion 获取文本回答
- 不附加 JSON schema hint
- 仍复用现有 provider/base_url/model/api_key 配置

## 前端设计

### 卡片结构

新增一个卡片区：

- 标题：`Chat with LLM (V0)`
- 副提示：说明当前仅基于单条测试索引问答

包含元素：
- 问题输入框
- 发送按钮
- 测试索引提示（当前使用的 doc_id / 显示名）
- 回答展示区
- 错误/状态提示区

### 建议文案

- 顶部提示：
  - `当前仅基于一条测试索引回答，用于先打通链路。`
- 无索引时提示：
  - `暂无可用索引，请先完成至少一条文献索引生成。`

### 前端状态

新增一个独立状态对象，避免污染现有上传/搜索逻辑：

```js
const chatV0State = {
  question: '',
  answer: '',
  loading: false,
  error: '',
  docId: '',
  displayName: '',
};
```

## 页面布局集成建议

在主页面现有结构中，Chat 卡片应位于：

- `.bottom-grid` 下方
- 与搜索区、预览区平级，不并入左右分栏

建议布局：

```html
<div class="main-layout">
  <section>上传区</section>
  <div class="bottom-grid">
    <section>搜索区</section>
    <section>预览区</section>
  </div>
  <section id="chatV0Section">Chat with LLM (V0)</section>
</div>
```

这样不会影响当前搜索/预览的左右布局逻辑，也更容易后续扩展成完整 Chat 区。

## 实施顺序

### Step 1：后端最小闭环

1. 新增 `chat_v0` prompt 文件
2. 新增 `chat_v0.py`
3. 新增 `generate_text()`
4. 新增 `chat.py` 路由
5. 在 `main.py` 挂载 chat router

### Step 2：前端最小卡片

1. 新增 Chat V0 卡片 UI
2. 新增发送函数
3. 调用 `/api/chat/ask_v0`
4. 展示回答与测试索引信息

### Step 3：联调与验证

验证以下情况：
- 有索引时可正常回答
- 无索引时明确报错
- provider 未配置时明确报错
- 回答仅基于测试索引，不编造

## 验收标准

- 页面上能看到 Chat V0 卡片
- 输入问题并点击发送后，能收到非空回答
- 页面能显示当前所用测试索引的 `doc_id` 和 `display_name`
- 无索引时有明确提示
- 不影响现有上传、搜索、预览功能

## 后续升级路径

V0 跑通后，按以下顺序升级最稳：

1. **V0-2**：把非流式改为流式
2. **V0-3**：从“单条测试索引”切到“多条索引拼接”
3. **V1**：正式引入会话、阶段进度、中断、引用证据

## 风险点

- 单条索引可能回答能力有限，但这是预期行为
- Markdown 全文可能包含格式噪音，若影响效果，后续可改成结构化纯文本上下文
- 若后续切全索引，必须先做上下文裁剪，否则容易超 token

## 结论

V0 应该优先追求：

- 极简
- 独立
- 易回退
- 快速验证 LLM 问答链路

因此本方案推荐：

- **单条测试索引**
- **单轮非流式回答**
- **独立卡片 UI**
- **独立 router / service / prompt 文件**

先打通，再扩展。
