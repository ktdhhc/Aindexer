# V3 重构方案

## 1. 文档目标

- 明确统一前端 V3 重写的整体方向。
- 对齐产品形态、前端技术栈、后端协作边界和分阶段迁移策略。
- 为后续开发提供一个可执行的实现基线，而不是继续在 V2 上原地演进。

## 2. 决策摘要

### 2.1 产品形态

- 这个产品应被视为一个本地桌面式工作台应用，而不是公开网站。
- UI 仍然应使用 Web 技术实现。
- 短期交付形态保持为：`FastAPI + 静态前端 + 本地数据/文件`。
- 后续可以再加上 `Tauri` 这类桌面壳，但这不应成为 V3 的第一步。

### 2.2 前端方向

- V3 应该是一个统一应用，把当前控制台、配置页和翻译工作流整合到一起。
- 不再延续当前 `多页 html + 原生 js + DOM 字符串模板` 的实现方式。
- 不在 `v2` 上原地重构，而是建立清晰的 V3 边界，再按功能逐步迁移。

### 2.3 后端方向

- 保留现有 `FastAPI` 应用、API 前缀、SQLite 和本地文件结构，作为 V3 初期的稳定基础。
- 仅在确实有助于 V3 组合能力和可维护性时，才重构后端内部实现。
- 在前端重写期间，对外公共 API 行为应尽可能保持稳定。

### 2.4 工作区（Workspace）边界

- 数据存储采用 Workspace 设计：每个 Workspace 是一套隔离的数据边界。
- 支持 Workspace 的新建、重命名和删除。
- 上传、解析、搜索、Chat 等主链操作都必须在指定 Workspace 中执行。
- Workspace 之间默认数据不可见，不允许跨 Workspace 混读结果。
- 默认保留系统工作区，确保历史数据可平滑迁移。

### 2.5 字段模板边界

- 字段配置从“单一全局字段表”升级为“字段模板体系”。
- 支持字段模板的新建、修改、删除，并支持基于已有模板复制创建。
- 索引生成时可选择字段模板，提取逻辑按选中模板执行。
- 模板内单个字段仍保持表单式编辑，但模板层采用卡片化管理。

## 3. 为什么不继续沿用现有前端技术栈

当前前端实际上已经分裂成两种实现风格：

- `backend/frontend/v2/`：控制台与配置页
- `backend/frontend/translator/`：翻译验证壳页面

已观察到的问题：

- 仪表盘逻辑集中在一个大型页面脚本中，渲染、状态、轮询、API 编排和 UI 行为混杂在一起。
- Shell 行为依赖 DOM id、全局变量和页面约定，而不是显式的组合关系。
- 页面样式体系不统一：部分页面使用 token CSS，部分页面依赖 Tailwind CDN 和内联样式。
- 控制台、配置页和翻译页没有真正作为一个应用构建，导致导航、状态和交互语言都是割裂的。
- 如果继续沿用这套实现方式，新增功能只会持续增加耦合度。

结论：

- 当前技术栈足以验证流程，但不足以支撑下一代长期维护版本。

## 4. 推荐的 V3 技术栈

## 4.1 核心栈

- `React`
- `TypeScript`
- `Vite`
- `Tailwind CSS`，但使用构建期配置，而不是 CDN 模式

原因：

- React 更适合构建统一的、操作密集型的工作台应用。
- V3 会把文献索引、Provider 配置、字段配置、聊天和翻译整合到同一个应用中，TypeScript 能明显降低复杂度。
- Vite 足够支持本地应用开发和静态产物输出。

## 4.2 路由与数据层

- `TanStack Router`：为统一应用提供类型化路由组织
- `TanStack Query`：负责服务端状态、轮询、缓存和请求生命周期处理
- `Zustand`：负责轻量本地 UI 状态，如 shell 状态、选区状态、面板状态和临时编辑器状态

原因：

- 这个项目有大量由服务端驱动的状态和异步流程。
- 现有的手动轮询和手动状态分发，应被统一的数据层替代。
- 基于路由的组织方式，能让控制台、配置和翻译在同一个 shell 内协同工作，同时保留清晰的功能边界。

## 4.3 表单与校验

- `React Hook Form`
- `Zod`

原因：

- Provider 配置和字段配置本质上都是表单密集型管理页面。
- 让校验规则贴近表单契约，可以减少重复页面逻辑。

## 4.4 UI 基础层

- 自定义设计 token
- 一层小而稳定的共享组件
- 如有需要，仅引入 `Radix UI` 作为交互原语
- PDF 预览和文本选择直接使用 `pdf.js`

原因：

- 这是一个工具型产品，它需要强 shell 和一致的交互行为，而不是重量级通用组件库。
- 翻译功能对 PDF 预览、文本选择、高亮和侧栏行为有直接控制需求。

## 4.5 测试

- `Vitest`
- `React Testing Library`
- `Playwright`

原因：

- 单元测试和组件测试应覆盖状态转换、路由行为和表单逻辑。
- 端到端测试应覆盖核心用户流程：上传、运行索引、搜索、预览、编辑、Provider 配置和选区翻译。

## 5. V3 的产品定位

V3 应被视为一个统一的本地工作台应用，包含三个一等域：

1. 控制台：上传、索引、搜索、预览、Markdown 编辑、导出、备份、聊天
2. 配置：Provider 配置、字段配置、系统级偏好设置
3. 翻译：文档上传、PDF 预览、选区翻译、Provider 设置、翻译历史

同时，以上域都运行在同一个 Workspace 上下文中，Workspace 是跨页面共享的一级状态。

整个产品应让用户感受到“一个应用中的多个工作区”，而不是多个页面被硬拼接在一起。

## 6. V3 前端架构

## 6.1 应用形态

- 一个统一前端，挂载在新的边界下，例如 `/v3/`
- 一个共享的应用 Shell
- Shell 内按路由组织的功能区域
- 一套贯穿整个应用的设计系统和交互语言

建议的顶层路由分组：

- `/v3/workbench`
- `/v3/config`
- `/v3/chat`
- `/v3/translator`
- `/v3/settings`（可选，如果系统级偏好继续增长）

## 6.2 建议目录结构

```text
frontend-v3/
  src/
    app/
      router/
      providers/
      shell/
    pages/
      console/
      config/
      chat/
      translator/
    features/
      documents/
      indexing/
      search/
      preview/
      editor/
      chat/
      provider-config/
      field-config/
      translation/
      pdf-viewer/
    entities/
      document/
      provider/
      workspace/
      translation/
      field/
    shared/
      api/
      ui/
      lib/
      styles/
      hooks/
      schemas/
```

## 6.3 分层职责

- `app/`：应用启动、路由、全局 provider、shell
- `pages/`：仅负责路由级组合
- `features/`：面向用户的功能工作流
- `entities/`：领域模型、适配器、选择器、轻量领域辅助逻辑
- `shared/`：通用基础设施和可复用 UI

规则：

- 不能让路由页面变成新的 `dashboard.js`。
- 复杂行为必须放在 feature 模块和 entity 适配器中。

## 7. UI 与交互原则

V3 是一个产品工作台，因此界面应遵循以下原则：

- 操作优先：用户应能快速进入下一步操作。
- 低解释密度：不要在主界面中堆砌教学型文案。
- 统一 shell：控制台、配置和翻译应共享导航、状态语言和反馈模式。
- 统一工作区上下文：跨页面共享当前 Workspace，并在所有核心请求中显式传递。
- 统一模板上下文：控制台索引操作显式传递 `field_template_id`。
- 层级稳定：通过侧边栏或顶层导航，让每个工作区边界明确。
- 视觉系统克制但有意识：不要重复 V2 那种混杂的样式模式。
- 翻译页是正式工作区，不是一个孤立的实验页面。

## 8. 后端协作策略

## 8.1 首先应保持稳定的部分

在构建 V3 的过程中，以下部分应保持稳定：

- `FastAPI` 应用装配方式
- 现有 `/api/*` 路由前缀
- SQLite 和文件型持久化布局
- 翻译相关 schema 和核心翻译 API 行为
- 现有上传、索引、搜索、导出和 Provider 流程，除非 V3 暴露出明确缺口

Workspace 例外说明：

- 为满足数据隔离目标，主链接口允许新增 `workspace_id` 上下文字段/参数。
- 该扩展应保持向后兼容：未传 `workspace_id` 时回落到默认工作区。

## 8.2 值得在 V3 期间同步推进的后端重构

这些后端改动值得做，因为它们能提升可维护性，又不会强行打破大范围契约：

- 提取主应用和翻译模块共用的 Provider 配置逻辑
- 把当前索引路由中的编排逻辑拆成更薄的 router 和 service/job 层
- 将大的 repository 文件拆成更小的领域仓储文件
- 仅在 V3 明确受益时，增加少量聚合型接口
- 增加 Workspace 仓储与路由，建立“工作区 -> 文献 -> 索引/搜索/聊天”隔离链路

## 8.3 应延后的后端改动

以下事项不应阻塞 V3 前端启动：

- 大规模存储迁移
- Secret 存储的完整安全重构
- 大范围 API 重命名
- 深度调整翻译缓存/历史模型，除非 V3 的具体工作流明确要求

## 9. 迁移策略

## 9.1 核心原则

- 在 V2 旁边构建 V3，而不是压在 V2 上做。
- 在 V3 达到可用对等之前，保留 V2 作为回退路径。

## 9.2 分阶段计划

### 阶段 0：定义与基础设施

- 确认路由地图、设计 token、功能边界和共享 app shell
- 创建新的前端工程和静态集成路径
- 基于当前后端接口建立类型化 API client 和领域适配器

退出条件：

- V3 应用能在现有后端中启动，并成功调用现有 API。

### 阶段 1：共享 Shell 与导航

- 建立全局布局
- 建立路由结构
- 建立全局 toast、dialog、loading、error 和后端健康状态模式
- 建立全局 Workspace 选择器和上下文传递机制

退出条件：

- App shell 稳定到足以承载所有主要工作区。

### 阶段 2：优先迁移配置页

- 实现 Provider 配置页
- 实现字段配置页
- 实现 Workspace 管理（新建/重命名/删除）

原因：

- 这两类页面规模更小，但表单密度高。
- 它们非常适合先稳定表单模式、API 契约、校验逻辑和 shell 组合方式。

退出条件：

- Provider 配置、字段配置和 Workspace 管理在 V3 中可用。

### 阶段 3：控制台只读工作流

- 文档列表
- 搜索与筛选
- 预览与详情展示
- 如有需要，可补充基础仪表盘指标
- 以上能力都需满足 Workspace 级隔离

退出条件：

- 用户可以在 V3 中浏览、搜索并检查现有数据。

### 阶段 4：控制台写操作工作流

- 上传
- 运行索引
- 轮询/进度展示
- 取消/重置
- Markdown 编辑
- 导出与备份入口
- 以上能力都需在当前 Workspace 内闭环完成

退出条件：

- 控制台主流程在功能上可于 V3 中使用。

### 阶段 5：翻译工作区整合

- 文档上传与选择
- PDF 预览
- 搜索与高亮
- 选区翻译
- 翻译历史与 Provider 设置对齐

退出条件：

- 翻译功能在 V3 中作为正式路由存在，而不是一个独立前端孤岛。

### 阶段 6：收敛与退役

- 在验证功能对等后，移除只属于 V2 的重复逻辑
- 如有需要，保留临时回退路径
- 逐步退役废弃静态页面

退出条件：

- V3 成为默认前端入口。

## 10. 建议交付顺序

推荐顺序：

1. Shell 与路由骨架
2. Workspace 管理 + 配置页整合
3. Chat 页面
4. 控制台浏览/搜索/预览
5. 上传与索引编排
6. Markdown 编辑与导出操作
7. 翻译工作区
8. 最后做清理和后端内部重构跟进

这个顺序可以降低迁移风险，并避免一开始就重建状态最重的工作流。

## 11. 风险与控制

## 11.1 主要风险

- 在 React 应用中重新制造一个新单体
- 让控制台和翻译在 V3 中仍然半分离
- 忘记在某些 API 请求中传递 Workspace 上下文，导致数据串读
- 在前端第一阶段就把过多后端清理工作一起拉进来
- 在信息架构稳定之前就急于迁移视觉风格
- 过早构建过多泛化抽象

## 11.2 控制规则

- 按领域和路由拆分 feature
- 保持服务端状态管理集中化
- 客户端 store 只保存真正属于 UI 的状态
- 在真实复用出现之前，避免构建大型自定义组件系统
- 在 V3 Web 应用稳定前，不引入桌面壳
- 所有主链 API 统一走 Workspace-aware 客户端封装，避免页面自行拼接参数

## 12. 重构验收标准

当满足以下条件时，可以认为 V3 重构成功：

- 控制台、配置页和翻译页已在同一个统一 app shell 中可用
- Workspace 已支持新建、重命名、删除，且主链操作在 Workspace 内可完整闭环
- 不同 Workspace 的上传文献、搜索结果和 Chat 上下文默认互相不可见
- 前端不再依赖页面级 DOM 约定和全局 hook
- 核心工作流可以在不回退到 V2 的情况下完成
- 状态与异步协作通过一致的前端数据层处理
- 样式已统一到一套 token 和组件系统下
- 后端公共 API 在迁移期间整体保持稳定
- 代码库相比当前 V2 结构明显更易扩展

## 13. V3 第一阶段的非目标

- 在前端落地前，先重写所有后端接口
- 把整个产品改造成云端多用户系统
- 构建复杂插件架构
- 引入大型企业级 UI 框架
- 在统一 V3 应用尚未可用前，就先调整桌面打包方式

## 14. 紧接着的下一步

1. 创建 V3 前端工程骨架，并接入现有 FastAPI 静态挂载策略。
2. 冻结第一版路由地图和功能边界（含 Workspace 上下文）。
3. 定义 Workspace-aware 共享 API client 和领域适配器。
4. 从配置页（Provider/字段/Workspace）和 Chat 页开始实现。
5. 在同一个 shell 之上推进控制台与翻译工作区迁移。
