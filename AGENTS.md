# 适用范围

本文件适用于位于 `D:\1project\literature-indexer` 的整个仓库。

将其作为在本仓库中工作的编码 agent 的默认操作指南。

## 一、全局 Agent 原则

以下原则适用于整个仓库。  
如果本文件后续章节提供了更具体的项目规则，则以更具体的规则为准。

### 1. 编码前先思考

不要想当然。不要掩饰困惑。要把权衡摆到明面上。

在开始实现之前：

- 明确说明你的假设。如果不确定，就提问。
- 如果存在多种理解方式，把它们都说出来，不要默默选一种。
- 如果有更简单的方法，就直接指出来。必要时要敢于提出异议。
- 如果有不清楚的地方，就先停下来。说明到底哪里令人困惑，并提出澄清问题。

### 2. 简单优先

只写解决问题所需的最少代码。不做任何猜测性扩展。

- 不添加任何超出要求的功能。
- 不为一次性代码做抽象。
- 不加入未被要求的“灵活性”或“可配置性”。
- 不为极不现实的场景编写额外错误处理。
- 如果 50 行就能完成，就不要保留 200 行实现。

问问自己：一个优秀工程师会不会觉得这份实现过度复杂？  
如果答案是会，那就继续简化。

### 3. 外科手术式修改

只改必须改的地方。只清理由你自己改动引发的问题。

在修改现有代码时：

- 不要“顺手优化”邻近代码、注释或格式。
- 不要重构那些本来没坏的东西。
- 要匹配现有局部风格，即使你自己平时会写成另一种样子。
- 如果注意到无关的死代码，可以指出，但不要直接删除，除非被明确要求。

当你的改动制造出“孤儿内容”时：

- 删除那些因你的改动而变得未使用的 import、变量、函数。
- 不要删除原本就存在的死代码，除非用户明确要求。

检验标准：每一行修改，都应该能够直接追溯到用户请求。

### 4. 以目标驱动执行

先定义成功标准。然后持续推进，直到验证通过。

把任务转化为可验证的目标：

- “添加校验” → 先为非法输入编写测试，再让这些测试通过
- “修复这个 bug” → 先写或确认复现方式，再让复现失效
- “重构 X” → 保持行为不变，并通过有针对性的验证确认

对于多步骤任务，先给出一个简短计划：

1. [步骤] → 验证：[检查方式]
2. [步骤] → 验证：[检查方式]
3. [步骤] → 验证：[检查方式]

### 5. 开发与验证习惯

- 后端命令通常在 `backend/` 下执行。
- 根目录命令通常用于启动脚本、打包脚本和仓库级检查。
- 当前正式前端仍由 `backend/frontend/` 提供静态资源；V3 方案已确定，但 V3 代码尚未成为默认实现。
- 优先运行有针对性的测试，不默认全量重跑。
- 高成本集成检查不作为默认快速验证方式。

### 6. 代码风格约束

- 后端：路由层保持精简，业务逻辑尽量下沉到 service 或更清晰的领域边界。
- 前端：避免再走 `单页超大脚本 + DOM 约定 + 全局变量` 的 V2 继续扩张路线。
- 文档：保持中文为主，路径和术语与仓库实际结构保持一致。

## 二、项目总体背景

- 产品类型：本地桌面式文献索引与翻译工作台
- 核心目标：围绕文献上传、结构化索引、搜索预览、手工修订和翻译工作流提供一套本地可运行工具
- 主要价值：将文献处理、索引维护和翻译验证收拢到同一套本地系统中
- 当前定位：已有可运行 demo，正在为 React V3 重构做文档沉淀和架构收敛
- 当前边界：当前仓库不是云端多用户系统；`octto/` 是独立子项目，不与主应用混写

## 三、相对稳定的仓库结构

```text
literature-indexer/
|- AGENT.md                 # 根级默认工作说明书
|- docs/                    # 当前事实、前端设计、方案文档
|- backend/                 # 主应用工作区
|  |- app/                  # FastAPI 入口、路由、服务、仓储、DB
|  |- frontend/             # 当前静态前端（v2 + translator）

|- data/                    # 运行态数据边界，不作为源码结构依据
`- octto/                   # 独立 TypeScript 子项目
```

## 四、主流程与分支流程的稳定认知

### 主流程

- Provider 配置
- 上传文献文件
- 启动索引任务
- 通过列表/搜索查看结果
- 预览并按需编辑 Markdown
- 导出或备份

### 分支流程

- 翻译工作流通过 `/api/translation/*` 与 `/translator` 前端独立存在
- 当前前端存在两套静态实现：`backend/frontend/v2/` 和 `backend/frontend/translator/`
- 后续 V3 目标是统一控制台、配置页和翻译页，但当前事实仍以前述两套前端为准

## 五、文档目录与读取建议

### 默认优先读取

#### `docs/CURRENT_STATE.md`
- 用途：当前仓库的稳定事实快照
- 内容：当前架构、主链流程、关键入口、默认验证方式、可维护性热点
- 读取时机：每次新会话默认优先阅读

#### `docs/FRONTEND_DESIGN.md`
- 用途：V3 首版前端的稳定设计约束
- 读取时机：任务涉及前端页面、样式、布局、交互或组件边界时再读取

### 按需读取

#### `docs/plans/`
- 用途：存放重构方案、feature 方案和重要架构取舍
- 读取时机：需要历史背景、方案依据或实现计划时再读

## 六、推荐读取顺序

- 先读 `AGENTS.md`
- 再读 `docs/CURRENT_STATE.md`
- 再读与当前任务直接相关的代码和测试
- 涉及前端体验时再读 `docs/FRONTEND_DESIGN.md`
- 需要历史背景或方案依据时，再读 `docs/plans/` 中相关文档

## 七、使用原则

- 优先相信代码与 `docs/CURRENT_STATE.md` 中仍然有效的事实。
- 不把历史文档、旧任务记录或运行态目录当作默认知识源。
- 如果任务只涉及一个功能区，只读该功能区代码、测试和必要文档。
- 做文档修改时，保持分层：全局规则进 `AGENTS.md`，当前事实进 `docs/CURRENT_STATE.md`，稳定设计进 `docs/FRONTEND_DESIGN.md`，方案推演进 `docs/plans/`。

## 八、Agent 修改检查清单

- 修改前先阅读被触及模块及其附近测试。
- 在发明新的命名、状态或载荷结构前，先匹配现有术语和数据形状。
- 后端行为变更时，优先补充或更新有针对性的测试。
- 若改动影响异步任务、轮询、取消或状态同步，必须考虑竞争条件和过期响应。
- 如果改动改变稳定工作流或默认入口，应同步更新相关文档。

## 九、不确定时的处理方式

- 优先做最小、局部、可回退的改动，而不是大范围重写。
- 对不确定事实，先查代码、测试和当前文档，不把推测写成结论。
- 如果某项信息高度不稳定，不要写入默认上下文文档。
- 在正式实施代码改动前，优先完成并确认文档体系。
