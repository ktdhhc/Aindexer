# 适用场景

当项目满足以下任一条件时，使用本 skill：

- 已经完成项目初步方案，准备进入正式开发
- 已经有一版可运行 demo，需要沉淀项目事实与约束
- 需要让后续 coding agent 在新会话中快速接手项目
- 需要为项目建立一套“低噪音、可维护、可持续更新”的文档体系

本 skill 的目标不是写一堆介绍材料，而是建立一套能真正服务后续开发的分层上下文系统。

# 核心目标

为项目生成或重构以下文档体系：

- `AGENTS.md`
  - 作用：作为每次会话前完整注入给 agent 的全局上下文文件
  - 内容重点：全局开发规则、代码修改原则、推荐读取顺序、文档分层说明
- `docs/CURRENT_STATE.md`
  - 作用：记录当前项目仍然有效的“事实快照”
  - 内容重点：当前架构、主流程、关键字段、默认行为、关键入口、默认验证方式
- `docs/FRONTEND_DESIGN.md`
  - 作用：记录已经基本确定、不会频繁变化的前端设计与交互规范
  - 内容重点：前端定位、视觉原则、页面布局、核心组件、状态规范、交互边界
- `docs/plans/`
  - 作用：存放项目初步方案文档的目录，以及后续少量重要更新方案
  - 内容重点：面向实现的 feature 方案、重大改版方案、架构取舍说明
  - 不承担“当前真相”的职责

# 这套文档体系的基本原则

你必须遵守以下分层原则：

## 1. 把“全局规则”和“项目事实”分开

- `AGENTS.md` 负责定义 agent 的工作方式
- `docs/CURRENT_STATE.md` 负责定义项目当前真相
- 不要把大量项目现状硬塞进 `AGENTS.md`
- 不要把开发原则写进 `CURRENT_STATE.md`，除非它会直接影响实现判断

## 2. 把“当前事实”和“历史方案”分开

- `docs/CURRENT_STATE.md` 只写当前仍有效的事实
- `docs/plans/` 记录方案、推演、重要决策与阶段性更新
- 不要把历史讨论、权衡过程、废弃想法追加到 `CURRENT_STATE.md`

## 3. 把“稳定设计”与“运行现状”分开

- `docs/FRONTEND_DESIGN.md` 记录相对稳定的 UI/UX 规范
- `docs/CURRENT_STATE.md` 记录当前前端入口、结构现状、关键限制
- 设计规范不等于当前实现细节
- 当前实现细节不等于设计规范

## 4. 保持默认上下文尽量短

后续 agent 的默认最小阅读路径应当是：

1. 先读 `AGENTS.md`
2. 再读 `docs/CURRENT_STATE.md`
3. 再读直接相关的代码与测试
4. 只有任务涉及前端体验时才读 `docs/FRONTEND_DESIGN.md`
5. 只有任务需要历史背景或方案依据时才读 `docs/plans/` 中的相关文档

# 输入材料要求

在执行本 skill 时，优先从以下材料提取信息：

- 项目 README
- 已有 demo 或现有代码仓库结构
- 当前运行方式
- 后端/前端关键入口文件
- 已有设计说明、需求草案、PRD、任务拆解
- 与项目当前状态直接相关的测试、脚本、页面结构说明

如果已有旧文档，优先复用其中“仍然有效、可迁移、可稳定保留”的内容，不要机械复制项目专属细节。

# 输出要求

你要输出以下内容：

1. `AGENTS.md` 的完整正文
2. `docs/CURRENT_STATE.md` 的完整正文
3. `docs/FRONTEND_DESIGN.md` 的完整正文
4. `docs/plans/` 的文件夹
5. 如有必要，补充 1-3 个初始方案文档模板文件名建议

输出时要尽量做到：

- 可直接落盘
- 不依赖额外解释才能使用
- 不夹带无关的分析过程
- 不把“不确定推测”写成确定事实

# 执行步骤

## 第一步：识别项目成熟度

先判断项目处于哪一类：

### A. 方案刚确定，尚未开发
特点：

- 有需求和方案，但代码很少
- 文档比代码多
- 当前需要的是“落地前框架”

此时：
- `AGENTS.md` 要偏重开发规则与文档约定
- `CURRENT_STATE.md` 可以更短，只写已确定的架构与范围边界
- `FRONTEND_DESIGN.md` 可以更完整，作为后续实现约束

### B. 已有一版 demo
特点：

- 已有前后端或至少一个可运行原型
- 已经形成部分稳定流程、字段、目录、入口文件
- 当前需要的是“从 demo 提炼稳定事实”

此时：
- `AGENTS.md` 要加入代码阅读顺序和修改原则
- `CURRENT_STATE.md` 要沉淀当前稳定真相
- `FRONTEND_DESIGN.md` 要从现有实现中提炼已经确定、应尽量保持不变的设计规则

## 第二步：提取四类信息

你必须把材料分成四类再写，不允许混写：

### 1) 全局开发规则
写入 `AGENTS.md`

包括：
- 编码前思考方式
- 简单优先
- 外科手术式修改
- 验证习惯
- 文档读取顺序
- 不确定时如何处理

### 2) 当前稳定事实
写入 `docs/CURRENT_STATE.md`

包括：
- 当前项目一句话说明
- 当前阶段
- 范围边界
- 当前架构
- 主流程或主链
- 关键入口文件
- 默认验证方式
- 仍然有效的 canonical 字段、主状态、关键目录

### 3) 稳定前端规范
写入 `docs/FRONTEND_DESIGN.md`

包括：
- 前端定位
- 页面结构
- 视觉基调
- 状态展示规范
- 核心组件集合
- 关键交互边界
- 开发减负规则

### 4) 方案与演进
写入 `docs/plans/`

包括：
- 初版方案
- feature 方案
- 架构变更方案
- 重点更新方案

不包括：
- 日常小修补记录
- 临时调试过程
- 已失效但无人会再看的一次性讨论痕迹

## 第三步：删掉不该进默认上下文的内容

生成文档时，主动排除以下内容：

- 冗长历史过程
- 大量会议纪要原文
- 不稳定的临时 workaround
- 两周内大概率又会变化的试验性约定
- 纯讨论性分析
- 与当前任务无关的背景介绍
- 面向展示的宣传文案

# 各文件的写作规则

## 一、AGENTS.md 写作规则

`AGENTS.md` 是给 agent 的“默认系统工作说明书”。

它必须满足：

- 内容尽量稳定，不因小功能迭代频繁改动
- 以工作原则、读取顺序、项目边界为主
- 不塞入大量实现细节
- 必须包含完整的全局开发规则模板
- 开头的全局原则内容必须原封不动地复制进新文档里

### AGENTS.md 必须包含的模块

1. 适用范围
2. 全局 Agent 原则
3. 项目总体背景
4. 相对稳定的仓库结构
5. 主流程 / 分支流程的稳定认知
6. 文档目录与读取建议
7. 推荐读取顺序
8. 使用原则
9. Agent 修改检查清单
10. 不确定时的处理方式

### AGENTS.md 禁止出现的内容

- 大量接口字段细节
- 每个 feature 的完整方案
- 长篇历史变更记录
- 高频变化的状态表
- 临时 bug 清单

## 二、docs/CURRENT_STATE.md 写作规则

`CURRENT_STATE.md` 是“当前真相快照”，不是历史日志。

它必须满足：

- 短，总长度控制在100-250行以内
- 稳，精准描述项目结构，不易随阶段内更新漂移
- 可覆盖更新
- 优先服务新会话快速接手

### 必须写什么

- 当前仍然有效的默认事实
- 当前主流程 / 主链
- 当前关键字段或关键结构
- 当前入口文件
- 当前默认运行方式
- 当前默认验证路径
- 当前可维护性热点
- 后续会话最容易误判的地方

### 不应写什么

- 历史过程
- 讨论痕迹
- 已废弃方案
- 一次性修复说明
- 局部实现细节
- 高频变化的调试结论

### 更新方式

- 只做覆盖式更新
- 删除已失效事实
- 不追加 changelog
- 不保留“曾经如此”的说明

## 三、docs/FRONTEND_DESIGN.md 写作规则

它是“稳定设计约束文档”，不是前端当前代码说明。

它必须回答的是：

- 前端应该长什么样
- 应保持什么交互语言
- 什么是首版必须守住的设计边界
- 什么复杂度不能引入

### 必须包含的内容

- 文档目的
- 前端定位
- 总体设计原则
- 视觉基调
- 颜色、排版、间距规范
- 页面布局规范
- 组件最小集合
- 关键页面规范
- 状态展示规范
- 表格 / 表单 / evidence / 差异展示规则
- 动效与响应式约束
- 开发减负规则
- 验收标准

### 不要写进来的内容

- 某个组件当前有哪些 bug
- 某一页的具体接口实现方式
- 高频变化的接口字段
- 当前页面状态管理的偶然技术细节

## 四、docs/plans/ 写作规则

`docs/plans/` 是方案目录，不是杂物箱。

适合放入的文档：

- 项目初版方案
- feature 设计方案
- 核心流程改版方案
- 数据结构改版方案
- 前端大调整方案
- 重大 prompt / workflow 调整方案

不适合放入的文档：

- 日常工作记录
- 每次小修改的说明
- 临时排障过程
- 与当前项目已无关的废弃草案

### docs/plans/ 推荐结构

- `docs/plans/README.md`
- `docs/plans/initial-project-plan.md`
- `docs/plans/<feature-name>.md`
- `docs/plans/<major-update-name>.md`

命名建议：

- 使用清晰、语义化、面向主题的英文 kebab-case 文件名
- 不默认用日期开头，除非这个项目强依赖版本/阶段时间线
- 文件名优先表达“日期+内容命名”

# 输出风格要求

- 面向工程执行
- 用词稳定、克制、可复用
- 不写空泛口号
- 不写“未来也许可能会”
- 不假装知道未知事实
- 能写成规则就不要写成散文
- 能写成短句就不要写成长篇解释

# 生成时的判断标准

在落笔前，始终判断：

- 这条内容是“全局规则”还是“项目事实”？
- 这条内容是“当前真相”还是“历史方案”？
- 这条内容是“稳定设计”还是“当前实现细节”？
- 这条内容会不会让后续 agent 多读很多无关信息？
- 这条内容半年后是否大概率仍然值得保留？

如果答案是否定的，就不要写进默认文档。

# 模板一：AGENTS.md

```md
# 适用范围

本文件适用于位于 `<repo-root>` 的整个仓库。

将其作为在本仓库中工作的编码 agent 的默认操作指南。

## 一、全局 Agent 原则（原封不动复制）

以下原则适用于整个仓库。  
如果本文件后续章节提供了更具体的项目规则，则以更具体的规则为准。

### 1. 编码前先思考

不要想当然。不要掩饰困惑。要把权衡摆到明面上。

在开始实现之前：

- 明确说明你的假设。如果不确定，就提问。
- 如果存在多种理解方式，把它们都说出来，不要默默选一种。
- 如果有更简单的方法，就直接指出来。必要时要敢于提出异议。
- 如果有不清楚的地方，就先停下来。说明到底哪里令人困惑，并提出澄清问题。

### 2. 简单优先

只写解决问题所需的最少代码。不做任何猜测性扩展。

- 不添加任何超出要求的功能。
- 不为一次性代码做抽象。
- 不加入未被要求的“灵活性”或“可配置性”。
- 不为极不现实的场景编写额外错误处理。
- 如果 50 行就能完成，就不要保留 200 行实现。

问问自己：一个优秀工程师会不会觉得这份实现过度复杂？  
如果答案是会，那就继续简化。

### 3. 外科手术式修改

只改必须改的地方。只清理由你自己改动引发的问题。

在修改现有代码时：

- 不要“顺手优化”邻近代码、注释或格式。
- 不要重构那些本来没坏的东西。
- 要匹配现有局部风格，即使你自己平时会写成另一种样子。
- 如果注意到无关的死代码，可以指出，但不要直接删除，除非被明确要求。

当你的改动制造出“孤儿内容”时：

- 删除那些因你的改动而变得未使用的 import、变量、函数。
- 不要删除原本就存在的死代码，除非用户明确要求。

检验标准：每一行修改，都应该能够直接追溯到用户请求。

### 4. 以目标驱动执行

先定义成功标准。然后持续推进，直到验证通过。

把任务转化为可验证的目标：

- “添加校验” → 先为非法输入编写测试，再让这些测试通过
- “修复这个 bug” → 先写或确认复现方式，再让复现失效
- “重构 X” → 保持行为不变，并通过有针对性的验证确认

对于多步骤任务，先给出一个简短计划：

1. [步骤] → 验证：[检查方式]
2. [步骤] → 验证：[检查方式]
3. [步骤] → 验证：[检查方式]

## 二、个性开发习惯（可随项目具体要求改变）

### 1. 开发与验证习惯

- 后端命令通常在 `backend/` 下执行。
- 前端命令通常在 `frontend/` 下执行。注意，前端文件避免完全写在同一个文件中，导致后期文件过大难以修改。更好的做法是在增加新组件或新功能时先思考：新建一个文件会不会更好？
- 优先运行有针对性的测试，而不是默认做全量重跑。
- 高成本集成检查不应作为默认快速验证方式。

### 2. 代码风格约束

- 后端：
  - 使用当前项目约定的语言版本与类型标注风格
  - API 路由保持精简，业务逻辑放在 service 层
- 前端：
  - 使用当前项目约定的组件风格与格式规范
  - 网络请求使用统一的异步与错误处理方式
  - 派生状态与请求协调遵循当前项目既有模式

这些原则偏向谨慎而不是速度。  
对于非常简单的任务，可以自行判断，不必机械执行。

## 三、项目总体背景

- 产品类型：`<一句话描述项目类型>`
- 核心目标：`<项目主要解决什么问题>`
- 主要价值：`<为什么这个项目存在>`
- 当前定位：`<内部工具 / demo / 产品原型 / 正式产品>`
- 当前边界：`<明确不做什么>`

## 六、文档目录与读取建议

### 默认优先读取

#### docs/CURRENT_STATE.md
- 用途：当前仓库的“当前真相”快照
- 内容：当前有效主流程、关键字段、默认行为、关键入口、默认验证方式
- 读取时机：后续 agent 会话默认优先阅读

#### docs/FRONTEND_DESIGN.md
- 用途：前端页面结构、视觉基调、交互边界与组件实现方向的规范说明

### 按需读取

#### docs/plans/
- 适合在实现某个具体 feature 或回看设计方案时按需读取
- 其他补充材料
- 适合在需要更完整背景时阅读，不应作为默认首读材料

## 七、推荐读取顺序

- 对于大多数日常开发任务，推荐使用下面的最小读取顺序：
 - 先读 docs/CURRENT_STATE.md
 - 再读与当前任务直接相关的代码文件
 - 再读相关测试文件
 - 只有在涉及前端表现时再读 docs/FRONTEND_DESIGN.md
 - 只有在需要历史背景或方案依据时，再读 docs/plans/ 中的相关文档

## 八、使用原则

- 优先相信代码和 docs/CURRENT_STATE.md 中仍然有效的事实。
- 不要把历史文档和方案文档当作默认最小上下文。
- 如果当前任务只涉及一个功能区，只读这个功能区的代码、测试和必要文档。
- 保持低上下文、强边界、局部验证的开发方式，避免无必要的全仓重读。

## 九、Agent 修改检查清单

- 修改行为前，先阅读被触及模块及其附近测试。
- 在发明新的命名和载荷结构之前，先匹配现有命名和数据形状。
- 对后端行为改动，更新或补充有针对性的测试覆盖。
- 先运行最小必要验证，再逐步扩大检查范围。
- 如果改动影响异步任务、轮询或状态同步，要考虑竞争条件和过期响应问题。

## 十、不确定时

- 优先做最小、局部的改动，而不是大范围重写。
- 保持行为可追溯、关键证据可保留、流程可恢复。
- 保持文档与行为一致；如果重大工作流假设发生变化，更新 docs/CURRENT_STATE.md 或相关文档。

# 模板二：docs/CURRENT_STATE.md

```md
# CURRENT_STATE

## 维护规则

- 应写什么：当前仍然有效且会直接影响开发判断的默认事实，如主链流程、关键字段、默认运行方式、关键入口文件、默认验证路径。
- 不应写什么：历史过程、讨论痕迹、未落地方案、一次性 workaround、局部实现细节、两周内很可能再次变化的临时约定。

- 更新方式：本文件只做覆盖式更新，不追加历史；删除已失效事实，保留当前真相。
- 防漂移要求：每次更新都应尽量短，只保留“新 agent 不知道就容易做错”的内容；若一条信息不能稳定维持一段时间，宁可不写入本文件。
