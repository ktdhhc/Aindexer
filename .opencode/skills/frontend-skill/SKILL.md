# Frontend skill

当作品质量更多取决于艺术指导、层级、克制感、图像运用与动效，而不是组件数量时，请使用这个 skill。

目标：做出让人感觉经过深思熟虑、具有高级感、并且符合当下审美的界面。默认朝着“获奖级”构图靠拢：一个鲜明的大想法、强有力的图像、精简的文案、严谨的留白，以及少量但令人记住的动效。

同时，本 skill 有一个额外前提：

- 对品牌型 / 展示型页面，强调构图、品牌存在感、图像叙事与动效节奏
- 对工具型 / 应用型页面，强调低文案、操作优先、强层级、少解释、少装饰

不要把“让界面看起来完整”误解为“往界面里加更多说明文字”。

## Mode Selection

开始前，先判断当前任务属于哪一类：

### A. Brand / Landing / Editorial 模式

适用于：

- 落地页
- 品牌页
- 宣传页
- 展馆 / 场馆 / 活动页
- 视觉 demo
- 叙事型单页

优先目标：

- 强视觉锚点
- 强品牌识别
- 海报感首屏
- 图像叙事
- 克制而可感知的动效

### B. Product / App / Workspace 模式

适用于：

- dashboard
- 管理后台
- 搜索页
- 数据工作台
- 配置台
- 表单驱动界面
- 高频操作工具

优先目标：

- 用户快速完成任务
- 低文案
- 操作比说明更突出
- 标题、状态、按钮一眼可扫
- 不把文档、教程或设计解释塞进主界面

如果任务本质是操作型产品界面，默认使用 Product / App / Workspace 模式，不要擅自套用落地页 hero 逻辑。

## Working Model

开始构建前，先写下四件事：

- 模式判断：这是 Brand / Landing，还是 Product / App
- 视觉命题：用一句话描述整体的情绪、材质感和能量感
- 内容计划：这一页由哪些 section 构成，每个 section 只做什么
- 交互命题：2–3 个会改变页面气质或强化操作感的动效想法

每个 section 只承担一个任务，只保留一个主导性的视觉想法，以及一个主要信息点或行动目标。

对于 Product / App 模式，还必须额外写下：

- 用户在这个页面的核心任务是什么
- 完成该任务最少需要哪些文字
- 哪些说明内容必须移出 UI，放到 README、帮助页或文档中

## Beautiful Defaults

- 从构图开始，而不是从组件开始。
- 优先使用全幅 hero，或占满画布的视觉锚点。
- 让品牌名或产品名成为最醒目的文字。
- 文案要短到能在几秒内扫完。
- 在添加装饰性 UI 外壳之前，先用留白、对齐、比例、裁切和对比度解决问题。
- 限制系统复杂度：默认最多两种字体、一个强调色。
- 默认采用无卡片布局。优先使用 section、分栏、分隔线、列表和媒体块。
- 把首屏当作一张海报，而不是一份文档。
- 当界面是工具型产品时，把主界面当作“工作表面”而不是“介绍页”。
- 能用布局表达的，不要用段落解释。
- 能用标签解决的，不要用句子；能用句子解决的，不要用段落。

## Minimal UI Discipline

当任务是 Product / App / Workspace 模式时，以下规则自动生效：

### 1. 只保留必要文案

界面中只允许保留那些对用户完成以下事项有直接帮助的文字：

- 理解当前页面
- 识别可执行操作
- 输入必要数据
- 查看状态
- 处理错误
- 做出决策

如果一段文字删掉后不影响任务完成，就删掉。

### 2. 默认不要解释性模块

除非用户明确要求，否则不要加入：

- 功能说明模块
- “这个功能如何工作”的区域
- onboarding 引导文案
- tips 提示面板
- helper sidebar
- best practices 卡片
- note 说明框
- FAQ 模块
- 空状态长文案
- 带描述的大段 hero
- 开发者解释性文字
- 设计评论式文字

### 3. 默认不要为“完整感”补文案

不要因为页面看起来太空，就补说明文字。

干净留白通常优于过度解释。

### 4. 文案预算

- 首屏默认不放大段描述
- 首屏描述性文字最多 1–2 行；能没有就没有
- 每个功能区优先只保留标题、状态、必要摘要、操作
- 空状态最多一句话，必要时再给一个动作
- toast 只反馈结果，不做教学
- 弹窗只解释当前动作所必需的信息

## Landing Pages

默认顺序：

1. Hero：品牌或产品、核心承诺、CTA，以及一个主导性的视觉元素
2. Support：一个具体功能、优惠，或证明点
3. Detail：氛围、工作流、产品深度，或故事
4. Final CTA：促成转化、开始使用、访问，或联系

Hero 规则：

- 只能有一个构图中心。
- 使用全幅图片，或主导性的视觉平面。
- 经典的 full-bleed 规则：在品牌型落地页中，hero 本身必须铺满到页面边缘，不能继承页面边距、外框容器或共享的最大宽度；只约束内部文字/操作列的宽度。
- 第一优先级是品牌，第二是标题，第三是正文，第四才是 CTA。
- 默认不要 hero 卡片、数据条、logo 云、药丸标签堆，或漂浮式 dashboard。
- 桌面端标题大约控制在 2–3 行；移动端应做到一眼可读。
- 文本列应保持窄而稳定，并锚定在图像中较为平静的区域。
- 所有叠加在图像上的文字都必须具备足够对比度，并且点击区域清晰可用。

如果首屏无法在脑海里被想象成一张海报，那它就不够强。

布局纪律：

- 首屏必须在常见笔记本尺寸下完成，不要依赖用户滚动才能看到主要信息。
- 如果首屏包含固定 / 粘性 header，那么这个 header 也要算进 hero 的空间预算里。header + hero 的总内容必须能放进桌面和移动端的初始视口。
- 当使用 `100vh` / `100svh` 的 hero 时，要减去持续存在的 UI 外壳高度（例如 `calc(100svh - header-height)`），或者把 header 叠加上去，而不是在正常文档流里向下堆叠。

## Apps

默认采用类似 Linear 的克制风格：

- 平静的表面层级
- 强有力的字体与间距
- 少量颜色
- 信息密集但易读
- 极少的装饰性外壳
- 只有当“卡片本身就是交互单元”时才使用卡片

对于应用型 UI，应围绕以下结构组织：

- 主工作区
- 导航
- 次级上下文或检查器面板
- 一个清晰的强调色，用于动作或状态

避免：

- dashboard 式卡片马赛克
- 给每个区域都加粗边框
- 在常规产品 UI 后面铺装饰性渐变
- 多个彼此竞争的强调色
- 不能提升扫读效率的装饰性图标
- 为了“教育用户”而塞入大段说明区
- 在列表、表格、表单上方堆叠介绍性面板
- 让 README 风格内容进入主界面

如果一个面板去掉卡片外观之后并不会失去意义，那就去掉卡片样式。

如果一个区域删掉说明段落之后仍然可用，就删掉那段说明。

## Imagery

图像必须承担叙事功能。

- 对于品牌、场馆、编辑型页面和生活方式产品，至少使用一张强而真实的图像。
- 优先使用场景内摄影，而不是抽象渐变或虚假的 3D 物体。
- 选择或裁切图像时，要保留一个适合放文字的稳定明度区域。
- 不要使用自带招牌、logo 或排版噪音、并与 UI 争抢注意力的图像。
- 不要生成内建 UI 外框、分屏、卡片或面板结构的图像。
- 如果需要多个时刻，就用多张图，而不是拼贴成一张。

首屏必须有一个真实的视觉锚点。装饰性纹理本身并不够。

对于 Product / App 模式：

- 不要为了“高级感”强行加入与任务无关的大图
- 不要让图像压过数据、表单、状态或核心操作
- 没有明确叙事职责的图像就不要加

## Copy

- 用产品语言写作，而不是写设计说明。
- 让标题本身承担主要意义。
- 辅助文案通常应是一句简短的话。
- 删掉 section 之间的重复表达。
- 不要把提示词语言或设计评论写进 UI。
- 每个 section 只承担一种职责：解释、证明、深化，或转化。

如果删掉 30% 的文案后页面更好了，那就继续删。

对于 Product / App 模式，再额外执行以下规则：

- 只保留任务完成必需的文案
- 页面不是教程，不要自发写“如何使用”
- 页面不是介绍页，不要自发写“这个功能是什么”
- 页面不是设计说明书，不要写“为什么这样设计”
- 说明内容应转移到 README、帮助页或独立文档
- 段落默认可疑；能不用就不用

## Utility Copy For Product UI

当作品是 dashboard、应用界面、管理工具或操作型工作台时，默认使用功能型文案，而不是营销型文案。

- 优先传达定位、状态和操作，而不是承诺、氛围或品牌腔调。
- 从工作界面本身开始：KPI、图表、筛选器、表格、状态，或任务上下文。除非用户明确要求，否则不要加入 hero section。
- section 标题应直接说明这是哪个区域，或用户可以在这里做什么。
- 好的例子："Selected KPIs"、"Plan status"、"Search metrics"、"Top segments"、"Last sync"
- 避免在产品界面中使用理想化 hero 文案、隐喻、campaign 式语言，或高管摘要式 banner，除非用户明确要求。
- 辅助文案应当用一句话说明范围、行为、新鲜度，或决策价值。
- 如果一句话看起来像首页 hero 或广告中的文案，就继续改写，直到它听起来像真正的产品 UI。
- 如果某个 section 不能帮助用户操作、监控或决策，就删掉它。
- 自检问题：如果一个操作人员只扫一眼标题、标签和数字，能否立刻理解这个页面？

再额外执行：

- 不要写会让人联想到帮助中心、教程页、产品宣传页的文案
- 不要给每个字段、卡片、表格都附解释性副文案
- 不要用长句包装简单状态
- 不要加入 “What this does” / “How it works” / “Tips” / “Notes” 一类区域
- 好的产品文案应该让用户立即操作，而不是先阅读

## Form / List / Empty State Rules

当任务包含表单、列表、表格、搜索或数据区时，默认遵守：

### Forms

- 只保留字段标签、必要时的 placeholder 与校验信息
- 不要给每个字段都附加解释段落
- 校验文案必须直接

推荐风格：

- 请输入 Base URL
- API Key 不能为空
- 格式不正确

不推荐风格：

- 此字段用于帮助系统识别你的接口地址配置
- 请输入有效密钥，以确保后续功能正常运行

### Lists / Tables

- 优先展示核心信息与直接操作
- 不要在列表顶部堆叠说明性卡片
- 不要在每一行加重复解释性副文案
- 行级操作必须直接

### Empty States

- 最多一句话
- 优先说明“当前没有内容”和“下一步动作”
- 不要写成长段引导

推荐风格：

- 暂无记录
- 没有搜索结果
- 还没有添加 API

## Motion

使用动效是为了建立存在感和层级，而不是制造噪音。

对于以视觉表现为主导的作品，至少交付 2–3 个有意图的动效：

- hero 中的一段入场序列
- 一个与滚动关联、sticky，或纵深感相关的效果
- 一个能强化可操作性的 hover、reveal，或布局过渡

如果可用，优先用 Framer Motion 来实现：

- section reveal
- 共享布局过渡
- 与滚动联动的透明度、位移或缩放变化
- sticky 叙事
- 用来推进叙事，而不只是填补空间的轮播
- 菜单、抽屉和模态框的存在感动效

动效规则：

- 在一段很短的录屏里也能看出来
- 在移动端要流畅
- 要快，而且克制
- 整页风格要一致
- 如果只是装饰，就删掉

对于 Product / App 模式，再补一条：

- 动效必须帮助扫读、反馈状态、强化可操作性或提升层级
- 不要用动效掩盖结构混乱或文案过多的问题

## Hard Rules

- 默认不要卡片。
- 默认不要 hero 卡片。
- 当需求要求 full bleed 时，不要做盒状 hero 或居中窄列 hero。
- 每个 section 不能有超过一个主导性想法。
- 不能依赖许多细小的 UI 装置去解释一个 section。
- 在品牌页面里，标题不能压过品牌本身。
- 不要写填充型文案。
- 除非文字位于平静、统一的一侧，否则不要做 split-screen hero。
- 没有明确理由时，不要超过两种字体。
- 除非产品本身已经有很强的系统，否则不要超过一个强调色。

对于 Product / App 模式，以下也是硬规则：

- 默认不要 hero section
- 默认不要功能介绍区
- 默认不要大段副标题
- 默认不要 FAQ、Tips、Best Practices、Notes 模块
- 默认不要 onboarding 文案塞进主页面
- 默认不要为了“完整”加入解释性段落
- 只要删掉不影响任务完成，就必须删
- README / docs 承担说明职责，UI 只承担操作职责
- 表单页只保留字段、按钮、校验提示
- 空状态最多一句话
- 非报错场景不要主动弹解释性 toast

## Reject These Failures

- 首屏给人的第一印象是泛化的 SaaS 卡片网格
- 图像很好看，但品牌存在感很弱
- 标题很强，但没有清晰行动
- 文字后面的图像过于杂乱
- 不同 section 重复同一种情绪性表述
- 轮播没有叙事目的
- 应用 UI 只是卡片层层堆叠，而不是清晰布局
- 工具型界面像介绍页，而不是工作界面
- 主界面里出现大段功能说明、使用说明或解释性文案
- README 风格内容被塞进页面
- 页面不是不好看，而是“话太多”

## Litmus Checks

- 在首屏里，品牌或产品是否一眼就能辨认？
- 是否存在一个强有力的视觉锚点？
- 只扫标题，能否理解页面？
- 每个 section 是否都只有一个任务？
- 这些卡片真的有必要吗？
- 动效是否真的提升了层级或氛围？
- 如果去掉所有装饰性阴影，这个设计是否仍然显得高级？

对于 Product / App 模式，还必须通过这些检查：

- 用户只看标题、标签、按钮和状态，能否立刻操作？
- 是否存在可以删除却不影响任务完成的文字？
- 是否把文档、教程或解释内容搬进了主 UI？
- 是否有任何区域只是为了“解释自己在做什么”而存在？
- 是否有任何 section 不能帮助用户操作、监控或决策？
- 首屏是否因为说明文字而变乱？
- 每个区域是否还能继续减字？

## Final Rule

品牌型页面要像海报，产品型页面要像工具。

不要把品牌页做成后台，也不要把后台做成说明书。
